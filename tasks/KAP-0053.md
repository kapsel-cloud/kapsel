# Prove The Isolated Live Sandbox

Status: queued; blocked by KAP-0052.

Problem: deterministic service tests cannot establish that real infrastructure isolates runs,
contains authority, survives runner failure, bounds capacity, and cleans owned resources. The public
site must not connect until those deployment properties are proven from an exact revision.

Direct owners: the sandbox API, deployment, threat, privacy, and testing contracts produced by
[KAP-0051](KAP-0051.md), the service package from [KAP-0052](KAP-0052.md), and umbrella
[KAP-0050](KAP-0050.md).

Depends on: [KAP-0052](KAP-0052.md). Completion allows the independent `kapsel.cloud` and Grafik
consumer packets to connect their fixture-proven adapters to the exact deployed interface.

## Result

One exact `kapsel-sandbox` revision runs both fixed scenarios on dedicated non-consequential
Kubernetes infrastructure with measured isolation, reconnect, saturation, restart, receipt,
retention, and forced-cleanup evidence. It is ready for bounded public traffic and independent
website integration.

## Work

1. Select infrastructure that explicitly supports the native Rust process, durable storage,
   networking, and Kubernetes isolation requirements. Record rejected alternatives and unknowns.
2. Provision a dedicated sandbox cluster and server-owned fixed images. Give each run a uniquely
   owned namespace, narrow service account, quotas, network restrictions, deadline, and cleanup
   identity.
3. Deploy durable admission, scheduler, runner, receipt storage, key custody, bounded observability,
   rate limits, concurrency cap, and global admission stop from reproducible configuration.
4. Prove healthy and unavailable-image runs, reconnect and replay, duplicate submission, saturation,
   expiry, runner restart at each owned seam, cleanup retry, and the global stop.
5. Prove that one run cannot read, mutate, project, or clean another run's resources or receipt.
6. Review public events, receipts, logs, metrics, traces, and retained artifacts for secrets,
   private identifiers, uncontrolled provider bodies, and unnecessary visitor data.
7. Measure startup, run duration, cleanup duration, resource use, retained bytes, and bounded cost
   at the configured concurrency ceiling.
8. Exercise deployment rollback and service unavailability without corrupting durable runs or
   promoting transport failure into a receiver result.
9. Publish only the exact versioned sandbox contract, service revision, bounded public endpoint, and
   approved technical facts needed by the website consumer.

## Acceptance

- Both fixed scenarios reach classifier-complete receipts through the public interface.
- Browser or client disconnect and runner restart recover the same operation without a blind second
  mutation.
- Cross-run isolation and ownership-safe cleanup pass adversarial tests.
- Saturation rejects before dispatch and reports bounded retry guidance.
- The global stop prevents new admission while preserving recovery and inspection of durable runs.
- Cleanup is forced after terminal, timeout, and abandoned-client paths and is independently
  observable from operation classification.
- Public output and retained operations data pass threat and privacy review.
- Deployment, rollback, live integration, and cleanup commands are reproducible from an exact clean
  revision.
- A residual-risk review explicitly approves bounded public traffic before the website points to the
  endpoint.

## Non-goals

No production workload, customer cluster, arbitrary request, account system, operator console,
billing, broad dashboard, second capability, managed-cloud product, or v1 support promise.

## Next action

After KAP-0052 passes deterministic acceptance, choose and document the smallest infrastructure that
can prove the owned isolation model; do not select a platform merely because the static website is
already deployed there.

## Residual risk

A public demonstration remains an abuse and operating-cost surface. Namespace isolation may be
insufficient against a compromised workload or cluster-level defect; stronger per-run isolation may
be required before traffic grows.
