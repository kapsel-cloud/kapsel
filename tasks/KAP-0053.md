# Prove The Isolated Live Sandbox

Status: planning accepted after independent final review of revision
`fe4ca1edb68dd527bf4c6de185beecd6f540bb2d` found no blockers. Gate 1 was separately authorized,
implemented in revision `eb62a92e30cfa77863de2d3cee305b4224db2a2c`, and accepted after independent
review found no remaining correction. Every live phase remains separately blocked behind the
deployment contract's explicit authorization gates. No provider is selected.

Problem: deterministic service tests cannot establish that real infrastructure isolates runs,
contains authority, survives runner failure, bounds capacity, and cleans owned resources. The public
site must not connect until those deployment properties are proven from an exact revision.

Direct owners: the sandbox API, deployment, threat, privacy, and testing contracts produced by
[KAP-0051](KAP-0051.md), the service package from [KAP-0052](KAP-0052.md), and umbrella
[KAP-0050](KAP-0050.md).

consumer packets to connect their fixture-proven adapters to the exact deployed interface.

## Planning record

The [deployment owner](../docs/SANDBOX_DEPLOYMENT.md#kap-0053-finite-deployment-experiment) now
records the finite provider-neutral experiment, reproduction lock, mandatory decision criteria,
surviving managed-Kubernetes options, ordered measurements, authorization gates, rejection rules,
and missing evidence. Current Kubernetes/provider statements cite official primary documentation.
The screen retains ordinary managed Kubernetes, GKE Sandbox, and AKS Pod Sandboxing as unranked
candidates. It rejects EKS on Fargate because AWS documents that the required VPC CNI NetworkPolicy
enforcement does not apply there and alternate CNIs are unavailable. It holds self-operated sandbox
runtimes out unless a managed candidate fails for a narrowly measured reason.

Planning found concrete execution prerequisites at the accepted base: `kapsel-sandbox` is a library,
not a native listener binary; its accepted deterministic service uses local SQLite and receipt paths
plus an in-process private digest key; and `Application` composition receives a raw Ed25519 receipt
seed. A later offline implementation revision must add only the fixed listener/operator control,
deployment fixture, evidence harness, durable volume/store and fenced backup/restore composition,
exact operator-owned patch admission, and raw-seed known-answer/stop fixture needed for this
experiment. Managed custody compatibility, IAM denial, audit, trust, outage, rotation, and
continuity remain live Gate 2/3 evidence. Non-export signing that cannot satisfy the existing
interface is a stop-and-review condition, not permission to redesign receipt semantics.

No account, credential, secret, provider resource, spend, DNS name, endpoint, deployment, or public
request was created or used in this planning wave. Consequently there is no current evidence of
isolation, custody, cleanup, restart, rollback, stop, saturation, bounded cost, or live
reproducibility, and the packet does not approve public traffic.

- A private HTTP `v1` adapter is permitted only alongside the native listener when exact
  request/response translation creates concrete pressure. It stays behind `handle_http` and owns

- KAP-0038 receipt-wire extraction is explicitly deferred and is not part of KAP-0053 unless a
  concrete defect requires it or an existing documented extraction trigger passes.

None is a separate packet or workstream. Any refactor used under these triggers must preserve
behavior and be validated through the existing interfaces.

## Gate 1 implementation evidence

The provider-neutral candidate adds one native `kapsel-sandbox` HTTP/1.1 process which incrementally
bounds raw request lines, headers, bodies, open connections, in-flight requests, and
receive/handling time before delegating exact route behavior to `Service::handle_http`. The same
binary exposes fixed local `stop` and `clear-stop` commands, not HTTP routes. Stop mutation now
requires one compatible singleton row and read-back; black-box restart tests prove stopped new
admission, retained replay, clear, and transport rejection without admission.

`deploy/sandbox/` locks the accepted service base, no-provider/non-public facts, container build,
private StatefulSet/static-volume shape, separate key roles, retention clocks, fenced backup/restore
sequence, and an exact normalized Deployment transition. The transition allows only the selected
named-container image and KAP-0038-required operation annotation under frozen username, namespace,
name, UID, owner, resource-version, current-image, target-digest, and operation preconditions. Its
adversarial matrix rejects runner identity, runtime, service-account, UID, owner, resource-version,
replica, sidecar, volume, label, security-context, annotation, operation, and digest changes.

The pinned Containerfile built locally without provider access as image ID
`sha256:08ced4f2100fd4eb956ec794fed6a46207b873870dab4738b542bc1b873a2675`; this is local build
evidence, not registry provenance or reproducibility proof. The raw-seed lane uses the RFC 8032
32-byte Ed25519 seed/public-key/signature known answer and then runs a receipt produced through the
existing `Application` into the production inspector under separate trust. The fixture explicitly
stops at managed-custody compatibility. The focused command is `cargo make test-sandbox-gate1`; it
creates no provider resource and uses no provider credential, cluster, endpoint, DNS, or public
traffic.

Independent review reproduced and corrected a projected-key mode mismatch, a raw-header one-read
overrun, insufficient workload-fixture assertions, and a missing real listener restart. Follow-up
review verified the fixes and found no blocker. The owning full deterministic gate and the separate
locked local image build passed.

## Result

One exact `kapsel-sandbox` revision runs both fixed scenarios on dedicated non-consequential
Kubernetes infrastructure with measured isolation, reconnect, saturation, restart, receipt,
retention, and forced-cleanup evidence. It is ready for bounded public traffic and independent
website integration.

## Acceptance

- Both fixed scenarios reach classifier-complete receipts through the public interface.
- Browser or client disconnect and runner restart recover the same operation without a blind second
  mutation.
- Cross-run isolation and ownership-safe cleanup pass adversarial tests.
- Saturation rejects before dispatch and reports bounded retry guidance.
- The global stop prevents new admission while preserving recovery and inspection of durable runs.
- Cleanup is forced only after a terminal report and receipt handoff, pre-attempt rejection, or
  setup failure durably known to precede `Application` invocation. Timeout and abandoned-client
  paths preserve receiver resources and drive same-operation recovery first; cleanup remains
  independently observable from operation classification.
- Public output and retained operations data pass threat and privacy review.
- Deployment, rollback, live integration, and cleanup commands are reproducible from an exact clean
  revision.
- A residual-risk review explicitly approves bounded public traffic before the website points to the
  endpoint.

## Residual risk

All live properties remain unproved. Gate 1's normalized patch model is not evidence that a real
Kubernetes admission implementation enforces the rule, and its static-volume/backup composition is
not evidence of storage-class support, snapshot atomicity, primary fencing, restore exclusion, or
managed key custody. The native process has no public-traffic approval and does not yet launch the
provider-dependent scheduler/runner, cleanup controller, or periodic retention loop; Gate 2 must
compose those existing operations under the selected workload, key, and storage identities. A public
demonstration remains an abuse and operating-cost surface; namespace policy may be insufficient
against a compromised workload or cluster-level defect, managed sandbox compatibility and key
custody may require interface review, cleanup may outlive the experiment, and fixed infrastructure
costs may exceed a future approved ceiling.
