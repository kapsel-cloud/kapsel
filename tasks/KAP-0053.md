## Gate 1 correction evidence

The unaccepted working-tree correction retains one native `kapsel-sandbox` HTTP/1.1 process which
incrementally bounds raw request lines, request heads, headers, bodies, open connections, in-flight
requests, and receive time before delegating exact route behavior to `Service::handle_http`. Its
30-second bound is only the connection's response wait and does not cancel continuing service work.
The same binary exposes fixed local `stop` and `clear-stop` commands, not HTTP routes. Those
commands now open only the existing private admission database, require one compatible singleton row
and read-back, and do not require receipt storage, tombstone-key availability, retention, or full
`Service::open`. Black-box restart tests prove stopped new admission, retained replay, clear, and
transport rejection without admission.

`deploy/sandbox/` now separates one API-owned system-state volume from one owner-private gateway
journal volume per active run. It includes an exact runner Pod/identity template, normalized
fail-closed journal-mount admission rule, the three contract-owned terminal journal deletion paths,
and a multi-volume backup-generation protocol that rejects incomplete or mixed generations. The API
workload mounts no gateway journal. These are offline topology and rule proofs only; real Kubernetes
admission, identity denial, storage fencing, snapshot consistency, restore, and retention remain
Gate 2/3 measurements. The existing exact Deployment transition still allows only the selected
named-container image and KAP-0038-required operation annotation under frozen preconditions.

8. Exercise deployment rollback and service unavailability without corrupting durable runs or
   promoting transport failure into a receiver result.
9. Publish only the exact versioned sandbox contract, service revision, bounded public endpoint, and
   approved technical facts needed by the website consumer.

## Acceptance

- Both fixed scenarios reach classifier-complete receipts through the public interface.
- Browser or client disconnect and runner restart recover the same operation without a blind second
  mutation.
- Cross-run isolation and ownership-safe cleanup pass adversarial tests.
- Saturation rejects before dispatch and reports bounded retry guidance.
- The global stop prevents new admission while preserving recovery and inspection of durable runs.
- Cleanup is forced only after a terminal report and receipt handoff, pre-attempt rejection, or
  setup failure durably known to precede `Application` invocation. Timeout and abandoned-client
  paths preserve receiver resources and drive same-operation recovery first; cleanup remains
  independently observable from operation classification.
- Public output and retained operations data pass threat and privacy review.
- Deployment, rollback, live integration, and cleanup commands are reproducible from an exact clean
  revision.
- A residual-risk review explicitly approves bounded public traffic before the website points to the
  endpoint.

## Residual risk

All live properties remain unproved. Gate 1's normalized patch model is not evidence that a real
Kubernetes admission implementation enforces the rule, and its static-volume/backup composition is
not evidence of storage-class support, snapshot atomicity, primary fencing, restore exclusion, or
managed key custody. The native process has no public-traffic approval and does not yet launch the
provider-dependent scheduler/runner, cleanup controller, or periodic retention loop; Gate 2 must
compose those existing operations under the selected workload, key, and storage identities. A public
demonstration remains an abuse and operating-cost surface; namespace policy may be insufficient
against a compromised workload or cluster-level defect, managed sandbox compatibility and key
custody may require interface review, cleanup may outlive the experiment, and fixed infrastructure
costs may exceed a future approved ceiling.
