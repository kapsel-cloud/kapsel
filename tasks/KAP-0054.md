# Specify The First Resident Gateway Pilot

Status: conditional backlog; blocked by KAP-0047 evidence, KAP-0046 direction, and one real pilot
workflow.

Problem: the accepted V1 technical direction names a customer-resident `kapseld`, but the daemon's
supported transport, process model, configuration, concurrency, upgrades, and compatibility promises
cannot be inferred safely from the public sandbox. A real workflow must select the smallest resident
interface before implementation begins.

Direct owners: [V1 technical direction](../docs/VISION.md), [technical scope](../docs/SCOPE.md),
[architecture](../docs/ARCHITECTURE.md), [testing](../docs/TESTING.md), and the future
pilot-specific technical owner created by this packet.

Depends on: approved aggregate evidence from [KAP-0047](KAP-0047.md), an explicit route from
[KAP-0046](KAP-0046.md), and a standalone publishable pilot objective. Private customer, buyer,
pricing, and company rationale remain outside this repository.

## Result

One evidence-selected pilot defines whether a `kapseld` package is justified and, if so, its
smallest supported customer-resident deployment, local interface, compatibility surface, integration
guide, and proof sequence.

## Decision work

1. Name the exact real workflow, existing operation, deployment environment, caller integration,
   operator authority model, recovery need, and support boundary using only approved technical
   facts.
2. Decide whether a resident daemon is necessary or whether the existing library, CLI, or MCP
   process already satisfies the workflow with smaller operational risk.
3. If a daemon is justified, define one-way `kapseld -> kapsel` dependency ownership for local
   admission, configuration, process lifecycle, health, bounded concurrency, upgrades, and
   diagnostics.
4. Select exactly one supported local transport and record idempotency, reconnect, cancellation,
   authorization, versioning, errors, and migration behavior.
5. Define installation, configuration, secret handling, credential and key rotation, backup,
   upgrade, rollback, downgrade, uninstallation, and incident boundaries.
6. Define the optional managed connection separately. Cloud disconnect must not corrupt, cancel,
   classify, or become the source of customer-resident execution truth.
7. Produce a finite implementation sequence and compatibility matrix. Reject every package
   extraction whose trigger in [V1 technical direction](../docs/VISION.md) has not passed.

## Acceptance

- Approved external-use evidence explains why a resident process is needed.
- The selected interface is tied to one real workflow and rejects broader speculative transports.
- Provider credentials and effect authority remain customer-resident by default.
- `kapseld` owns process and local integration behavior without duplicating effect lifecycle.
- The proposal names every compatibility promise and unsupported interface explicitly.
- Migration, restart, upgrade, rollback, concurrency, credential rotation, supported-platform, and
  pilot acceptance tests have finite owners.
- Package choices cite passed extraction triggers; future receipt, protocol, SDK, provider,
  Kubernetes, storage, and separate CLI packages remain design backlog when unearned.
- KAP-0046 records rejected alternatives and authorizes exactly one next implementation route.

## Non-goals

No standing daemon implementation before evidence, generic control plane, cloud-held customer
provider authority by default, multi-provider platform, arbitrary capability interface, broad SDK
program, or automatic v1 release authorization.

## Next action

Remain blocked. Revisit only after KAP-0047 and KAP-0046 provide approved technical direction and a
real pilot workflow requires a resident process.

## Residual risk

A single pilot can overfit the resident interface. Compatibility should be limited to the exact
workflow and support window until repeated integrations expose a stable repeated seam.
