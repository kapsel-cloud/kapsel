# Implement Replaceable Backend Submission

Problem: Accepted statements need reliable registration with one external transparency backend without coupling core evidence semantics to that operator or overstating network attempts.

Acceptance:

- Define the minimal interface: `register_statement`, `lookup_submission`, `retrieve_receipt`, and `retrieve_trust_metadata`.
- Implement one backend from KAP-0003 and preserve native receipt bytes.
- Persist durable internal sync state for pending, in-flight, ambiguous, retryable, acknowledged, and terminal outcomes.
- Recover duplicate and remote-success/local-crash cases idempotently using stable statement hash/idempotency and lookup.
- Handle timeout, ambiguity, rate limiting, and terminal rejection explicitly.
- Transition public state to `submitted` only after backend acknowledgement/submission ID; attempted or timed-out requests remain publicly `accepted`.
- Transition to `witnessed` only after a valid receipt is retrieved, persisted, and associated with the accepted statement.
- Expose oldest-unwitnessed age and never report registration as truth, completeness, or endorsement.
- Enforce an allowlist for submitted fields; send no arbitrary collateral, prompts, records, or secrets.
- Treat backend trust metadata as untrusted input until external policy anchors it.
- Keep backend code at the edge and document replacement implications.

Verification:

- Contract, outage, ambiguity, duplicate, recovery, privacy, state-transition, and offline-receipt tests pass.
- No attempted request creates a false remote fact.
- Integration review has no unresolved blocker/high finding.
