# Package And Document The V1 Release Candidate

Status: active technical cursor.

Problem: implemented commands and adapters are not a release until a fresh evaluator can install an
identified artifact, run the supported journey, understand failure states, and remove owned
resources.

Contract owners: [technical scope](../docs/V1.md), [build](../docs/BUILD.md),
[testing](../docs/TESTING.md), [security](../SECURITY.md), and [privacy](../docs/PRIVACY.md).

Depends on: [KAP-0043](KAP-0043.md).

## Work

- Name the supported Unix target set for V1.
- Produce a versioned artifact per supported target with SHA-256 checksums, license, and exact
  source revision.
- Add clean-environment install, version/provenance, operation, restart, inspection, MCP, and
  cleanup smoke tests.
- Publish evaluator setup, operator configuration, expected output, failure-state explanation,
  limits, and architecture diagrams.
- Document prototype-scoped signed formats, unsupported platforms, security boundaries, and every
  material non-claim.
- Define reproducible release-candidate assembly without creating a stable evidence-package or
  plugin contract.

## Acceptance

- A clean supported environment installs and runs only from the release-candidate artifacts and
  published documentation.
- Artifact digest, license, version, and source revision agree.
- Setup does not require repository-internal commands beyond the separately documented source-build
  path.
- Documentation distinguishes request acceptance, receiver result, `UNKNOWN`, inspection, and
  production-readiness limits.
- Artifact smoke tests cover the local command, offline inspector, MCP adapter, and owned demo.
- `cargo make fmt`, focused tests and Clippy, `./scripts/ci-local.sh`, the explicit live-kind gate,
  artifact smoke tests, local Markdown links, and `git diff --check` pass.

## Non-goals

No stable registry/library contract, automatic updater, Windows support without evidence, hosted
distribution service, or marketing launch. A pre-V1 crates.io alpha does not replace the owned V1
artifacts, checksums, source-revision proof, or smoke tests.

## Progress

The release-candidate contract names only `x86_64-unknown-linux-gnu`, built and smoked in pinned
x86-64 Debian 12 containers. Version `0.1.0-rc.1` assembly now produces one normalized archive and
adjacent SHA-256 file with exact source, target, builder, license, and binary provenance. The
archive keeps the ordinary binary separate from the feature-gated demo executable and includes one
standalone evaluator guide.

Artifact smoke verifies dirty-tree refusal, checksum, bounded exact layout, normalized modes,
provenance agreement, safe extraction, grant provisioning, operation and restart, offline
inspection, MCP discovery and call, both demonstration crash seams, frozen receipt bytes under
rotated settings, bounded output, and cleanup using only extracted files. Two isolated assemblies
produced byte-identical archive and checksum bytes. Hosted run
[`29588910295`](https://github.com/kapsel-cloud/kapsel/actions/runs/29588910295) passed the default,
artifact-smoke, and reproducibility lanes at revision `ab96fd0c583c5828e2c03740237f45cf7ec6c2c6`.
Push runs are now configured to follow those gates with one strict clean assembly and upload the
versioned archive plus adjacent checksum under a source-revision-named workflow artifact.

The local public live-kind demo was attempted twice after the artifact changes, but this Docker
Desktop environment did not make the kind control plane ready within the owner-defined 120-second
creation bound. The uniquely named clusters were removed manually after each failed attempt. The
harness now marks its unique cluster name owned before creation and has a deterministic regression
proving failed creation triggers deletion; a bounded diagnostic request prevents cleanup from
waiting indefinitely. KAP-0044 remains active until the explicit live-kind gate passes in a working
environment and a clean-checkout artifact records `source_dirty: false`.
