# Freeze Minimal Evidence Contracts And Vectors

Problem: Core proof objects, package cardinality, verdict aggregation, and offline trust inputs must be profile-oriented and unambiguous before implementation.

Acceptance:

- Define `ProfileId`, `EvidenceStatement`, `SignedStatement`, `TransparencyReceipt`, `EvidencePackage`, `TrustPolicy`, and `VerificationReport`.
- Freeze one-statement semantics: exactly one Kapsel-issued signed statement and one corresponding transparency receipt per witnessed V1 package.
- Define that the statement commits to the profile's authorization, action, and outcome material; externally signed authorization and receiver material are profile collateral.
- Keep the external trust policy outside the package.
- Embed all other required offline verification collateral as untrusted bytes; URL-only references are insufficient and backend-returned metadata is never trusted by provenance alone.
- Define package manifest, profile dispatch/versioning, disclosures, commitments, omissions, deleted material, and external trust semantics.
- Define exact V1 SCITT/COSE/CBOR subset from KAP-0003.
- Define verdict aggregation and exit semantics: `VERIFIED` only when every policy-required dimension passes; otherwise `FAILED`, `UNTRUSTED`, or `INCOMPLETE` with independent structure, cryptography, trust, witnessing, profile, disclosure, and completeness dimensions.
- Define hostile-input resource limits and unknown-critical-field behavior.
- Commit valid, altered, malformed, unsupported-profile, untrusted, unwitnessed, incomplete, and linkage-failure vectors.
- Keep authorization/action/outcome linkage primitives generic; profiles define domain semantics.

Verification:

- Core contracts contain no workflow-specific fields.
- Independent implementation can consume vectors without reading Rust code.
- Bundle-supplied collateral cannot establish trusted validity.
- No fixture or report can call a merely structural/self-consistent package `VERIFIED`.
