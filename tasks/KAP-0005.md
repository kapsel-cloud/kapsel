# Create Static Reference Fixture Matrix

Problem: The verifier should develop against static profile-bound packages before storage, adapters, or online services exist.

Acceptance:

- Create one `kapsel.reference-action.v1` fixture covering signed authorization, bounded action, stable correlation ID, receiver-derived outcome, and disclosure/commitment material.
- Use exactly one Kapsel signed statement; include profile collateral and one receipt for witnessed cases.
- Cover verified trusted, modified evidence/statement, wrong/unknown profile, unknown critical header, embedded self-trust, expired/revoked/wrong-purpose key, malformed receipt, missing/deleted disclosure, valid-unwitnessed, and linkage-failure cases.
- Add invalid cardinality fixtures for zero or multiple Kapsel statements and multiple conflicting receipts.
- Add bounded hostile package cases for traversal, duplicate paths/keys, bombs, excessive depth/size, malformed CBOR, downgrade, and conflicting claims.
- Keep fixture bytes deterministic and contractual.
- Keep profile-specific fixture generation outside core proof code.

Verification:

- Every mutation fails or reports the intended non-verified verdict.
- Fixtures contain no real customer data or secrets.
- Generic package fixtures remain usable when another profile is compiled in.
