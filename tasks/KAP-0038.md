# Release One Kubernetes Effect-Gateway Experiment

Status: active V1 program index; implementation cursor is KAP-0045.

Problem: Kapsel needs one executable proof that an autonomous agent can cross a bounded,
crash-recoverable effect gateway instead of receiving unrestricted Kubernetes authority.

Contract owners: [technical scope](../docs/V1.md),
[KAP-0038 experiment owner](../docs/experiments/KAP-0038-kubernetes-effect-gateway-boundary.md),
[architecture](../docs/ARCHITECTURE.md), [threat model](../docs/THREAT_MODEL.md),
[testing](../docs/TESTING.md), and [build](../docs/BUILD.md).

This task owns release-level acceptance and the evidence index. Ordered child packets own
implementation work. This task does not define operation, lifecycle, result, receipt, trust, or
recovery semantics.

## Implemented foundation

- One bounded `kubernetes.set_deployment_image` request grammar.
- Owner-signed fixed-purpose exact grants under application-configured trust that agent input cannot
  choose.
- Safe target validation before either durable `not_attempted` or atomic target identity plus
  `apply_started`, with fair transient-read deferral.
- Classifier-complete receipt V2 with offline recomputation and durable exact-byte preparation.
- SQLite rollback-journal durability with `synchronous=FULL`.
- Deterministic fault injection across mutation and finalization windows, including real subprocess
  kills and restarts at the mutation and receipt-publication seams.
- Healthy and unhealthy-image journal-reopen proofs against disposable `kind` resources; the
  unhealthy path observes `ProgressDeadlineExceeded`, emits a signed `FAILED` receipt, and inspects
  it offline.
- UID/resource-version-guarded, name-keyed strategic image mutation.
- Bounded receiver observation with `SUCCEEDED`, `FAILED`, or `UNKNOWN`.
- Prototype-scoped signed receipt bytes, canonical vectors, immutable Unix publication, and bounded
  offline inspection under separate trust, explicit time, and limits.
- A prototype local evaluator command for exact-grant provisioning, bounded operation or restart,
  and classifier-complete offline receipt inspection through the shared application interface.
- A release-owned disposable-kind demonstration that kills the real command process after one
  returned mutation and after frozen receipt publication, then proves one apply, failed-rollout
  classification, frozen bytes under rotated settings, offline inspection, and owned cleanup.
- A thin MCP `2025-11-25` stdio adapter with exactly one five-field tool, out-of-band operator
  startup configuration, bounded protocol lifecycle and errors, and typed-outcome equivalence with
  the local application adapter.

These are pre-V1 experiment surfaces, not a stable installed command or completed V1 release.

## Release acceptance

- A fresh evaluator can install the release artifact and run the healthy and crash/failed-rollout
  paths from documented commands without repository-internal sequencing.
- Agent-facing local and MCP requests contain only the bounded operation intent. Operator-owned
  grants, trust, Kubernetes credentials, signing material, journal/output paths, and demo fault
  control stay behind application composition.
- Operator grant provisioning is explicit, separate from request submission, and never treats a
  caller-supplied key or self-signed grant as authority.
- The release demo terminates a real process at both owner-required seams and recovers through the
  same public application interface used without faults.
- Malformed input or grant, untrusted authorization, tuple mismatch, and mutable image are rejected
  before persistence or provider calls. A missing Deployment, missing named container, or invalid
  target becomes terminal `not_attempted` before any attempt marker or patch. Transient target-read
  errors defer fairly and cannot block later authorized operations.
- Recovery after the durable mutation marker performs bounded observation and no blind second patch.
- Request acceptance and receiver outcome remain distinct in state, receipt, report, and demo.
- Offline inspection requires separate trust, explicit evaluation time, and explicit limits; it
  performs no network access and never reports `VERIFIED`.
- Secrets and unbounded Kubernetes response bodies never enter requests, SQLite, receipts, reports,
  errors, or captured logs.
- The local command and MCP adapter are thin conversions into the same gateway interface; neither
  reimplements validation, authorization, lifecycle, recovery, classification, or receipt logic.
- The demo creates and removes only resources it owns and preserves bounded failure logs.
- Release artifacts identify the exact source revision, include SHA-256 checksums and the license,
  and pass an install/run/inspect smoke test on every named supported target.
- A clean-checkout rehearsal passes `cargo make fmt`, focused tests and Clippy,
  `./scripts/ci-local.sh`, the explicit live-kind gate, command/MCP integration tests,
  release-artifact smoke tests, local Markdown links, and `git diff --check`.
- The V1 tag and release notes name supported platforms, commands, signed-format prototype status,
  claim limits, and residual risks without claiming production readiness.

## Latest explicit evidence

On 2026-07-16, KAP-0040 made the repository root the single `kapsel` product package, retained
`kapsel-dev` as unpublished development tooling, and added the request-only `Application`
composition interface. Operator configuration owns the exact grant, trust, Kubernetes client,
receipt authority, and private paths. Reconciliation and finalization select the configured
operation exactly even in a shared journal. Focused tests and Clippy, `./scripts/ci-local.sh`,
strict rustdoc, package-archive CI, the crates.io dry run, and
`./scripts/test-kind-effect-gateway.sh` passed. The live gate proved both healthy and failed-rollout
paths and deleted its owned cluster.

On 2026-07-17, KAP-0041 completed the prototype evaluator commands and reorganized the one product
package into a compact interface map, thin executable composition root, shared application module,
and private deep gateway tree. Public rustdoc items, canonical vectors, SQLite schema, provider-call
assertions, command behavior, simulation, fuzz, and live-kind semantics were preserved.

KAP-0042 added the release-owned public crash demonstration. Deterministic black-box tests and the
owned live-kind harness killed the feature-gated production command at the post-mutation and
post-publication seams. Recovery kept the provider apply count at one, classified the unavailable
image from `ProgressDeadlineExceeded`, preserved the frozen receipt under rotated settings, exposed
all signed classifier inputs through offline inspection, retained bounded failure diagnostics, and
removed every owned cluster.

KAP-0043 added the fixed-schema MCP `2025-11-25` stdio adapter without a generic SDK or new
dependency. Black-box process tests proved exact tool discovery, out-of-band operator authority,
local request and outcome equivalence, all bounded result classes, protocol lifecycle and framing,
hostile-input rejection, restart, and secret-free responses. The default deterministic repository
gate and isolated demo-harness gate passed.

KAP-0044 produced the reproducible `0.1.0-rc.1` x86-64 GNU/Linux artifact, adjacent checksum,
standalone evaluator guide, safe artifact smoke, and hosted downloadable workflow artifact. At exact
revision `623b1bb712a18abadd3749d53ce8b961aacf4c52`, hosted default, artifact, reproducibility, and
strict clean assembly lanes passed. A clean supported host then passed both live-kind proofs and the
source-built and downloaded-artifact real-process demonstrations; each removed its owned cluster.
