# Define The Public Sandbox Contracts

Status: next queued contract packet; KAP-0049 is complete.

Problem: the public sandbox introduces admission, scheduling, reconnectable projection, retention,
cleanup, abuse, and deployment concerns that the current gateway contracts do not own. Implementing
a Rust service before those interfaces are bounded would let transport or hosting accidentally
redefine effect lifecycle and receipt meaning.

Direct owners: [technical scope](../docs/SCOPE.md),
[active experiment boundary](../docs/experiments/KAP-0038-kubernetes-effect-gateway-boundary.md),
[V1 technical direction](../docs/VISION.md), [architecture](../docs/ARCHITECTURE.md),
[threat model](../docs/THREAT_MODEL.md), [privacy](../docs/PRIVACY.md),
[testing](../docs/TESTING.md), and new sandbox API and deployment owners created by this packet.

Depends on: [KAP-0049](KAP-0049.md). Feeds [KAP-0052](KAP-0052.md) and the umbrella
[KAP-0050](KAP-0050.md).

## Result

The repository contains complete contract owners for one fixed public sandbox operation before an
HTTP dependency, scheduler, or deployment implementation is added.

## Work

1. Amend technical scope and the active experiment boundary only enough to distinguish the fixed
   hosted sandbox composition from the release-owned local-kind demonstration. Name exactly which
   `Application`, lifecycle, result, and receipt semantics are reused, and move sandbox admission,
   projection, retention, and deployment behavior under new direct owners. Preserve the sole
   capability and all existing non-claims.
2. Define a versioned sandbox HTTP interface for fixed `healthy` and `unavailable-image` scenarios,
   durable run identity, admission responses, status snapshots or ordered event replay, terminal
   receipt retrieval, expiry, saturation, and bounded errors.
3. Define the public projection independently from gateway internals. Local paths, raw journal rows,
   fault controls, credentials, uncontrolled provider logs, and private runner identifiers are not
   protocol fields.
4. Define deployment ownership: edge admission when used, native Rust runner, durable admission
   store, bounded scheduler, dedicated sandbox cluster, per-run namespace and service account,
   receipt storage, key custody, and forced cleanup.
5. Extend threat and privacy owners for anonymous abuse, enumeration, denial of service, request
   correlation, event and receipt disclosure, retention, runner compromise, namespace escape,
   cleanup failure, dependency or image compromise, and kill-switch operation.
6. Extend testing with deterministic API and scheduler contracts plus live isolation, reconnect,
   saturation, timeout, runner restart, and cleanup-failure lanes.
7. Record the exact compatibility status of the sandbox interface. It remains demonstration-scoped
   and does not become the production `kapseld` interface.

## Acceptance

- Every public field has an owner, bound, disclosure rule, and versioning status.
- The caller can select only one of the two fixed scenarios and cannot appoint any authority,
  target, image, credential, lifecycle action, callback, or result.
- Admission durability and idempotency are distinct from gateway submission and provider outcome.
- Disconnect, timeout, stream failure, and cleanup status cannot become Kapsel receiver outcomes.
- Reconnect and replay ordering are explicit and do not expose internal gateway fault points.
- Namespace ownership, quotas, deadlines, concurrency, retention, cleanup, and the global admission
  stop have normative owners.
- Contract examples cover healthy, unavailable-image, saturation, expiry, unknown version, and
  unavailable service behavior.
- Focused link, format, terminology, and overclaim checks pass before KAP-0052 starts.

## Non-goals

No implementation, framework selection, hosting-provider commitment, production cluster, arbitrary
Kubernetes input, customer credential, login, operator console, fleet view, generic log API, second
capability, or production compatibility promise.

## Next action

## Residual risk

A complete contract can bound intended behavior but cannot prove that the selected infrastructure
actually provides the required isolation, cleanup, capacity, or cost control.
