# Implement Synthetic Reference Profile And Adapter

Problem: Technical V1 needs an architecture canary that cannot be blocked by or confused with commercial profile selection.

Acceptance:

- Implement `kapsel.reference-action.v1` by default; substitute a discovered workflow only if its contract is stable before implementation begins.
- Model signed authorization, bounded attempted action, stable idempotency/correlation ID, receiver-derived outcome, disclosure/commitment material, and meaningful reconciliation.
- Implement a small deterministic HTTP receiver or sandbox adapter at the side-effect boundary.
- Give the adapter an operation journal covering stable correlation/idempotency -> external call -> outcome observation/reconciliation -> finalized evidence.
- Document the crash window after external action but before Kapsel accepts the final signed statement.
- State that V1 does not prove authorization was witnessed before execution; multiple linked/pre-action statements are outside scope.
- Implement `EvidenceEncoder` and `ProfileValidator`; register them at compile-time composition roots.
- Keep profile and adapter dependencies inward; core/storage/verifier/transparency crates contain no reference-workflow fields.
- Prove authorization -> action -> receiver-derived outcome linkage and document what remains unproven.

Verification:

- End-to-end reference package verifies offline.
- Crash/reconciliation tests cover external success before final statement acceptance.
- Removing the profile/adapter leaves all core crates buildable and useful.
- KAP-0012 has no dependency on KAP-0016 commercial selection.
