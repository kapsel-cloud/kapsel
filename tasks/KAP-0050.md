# Prove One Live Sandbox Run

Status: queued contract and implementation packet; blocked by KAP-0049.

Problem: the local evaluator proves Kapsel to developers willing to provision its environment, but
the public product surface cannot run the proof. One hosted vertical slice must establish whether
the same Rust application can safely expose a fixed non-consequential demonstration without turning
the browser into an authority source or prematurely creating a generic control plane.

Direct owners before implementation: the [technical scope](../docs/SCOPE.md),
[architecture](../docs/ARCHITECTURE.md), [threat model](../docs/THREAT_MODEL.md),
[privacy boundary](../docs/PRIVACY.md), [command contract](../docs/COMMANDS.md),
[receipt contract](../docs/experiments/KAP-0038-kubernetes-effect-gateway-boundary.md), and a new
sandbox API/deployment owner created by this packet.

Depends on: [KAP-0049](KAP-0049.md). Feeds the evaluator evidence cycle in [KAP-0047](KAP-0047.md).
The public consumer is independently owned by the
[`kapsel.cloud` task route](https://github.com/kapsel-cloud/kapsel.cloud/blob/master/tasks/README.md).

## Result

A visitor can start one fixed `kubernetes.set_deployment_image` demonstration, follow the real
Kapsel lifecycle through reconnectable public status, and inspect the resulting signed receipt. The
operation runs on isolated non-consequential infrastructure with server-owned authority and forced
cleanup.

The hosted path reuses the same application semantics as the local CLI and MCP adapter. It does not
introduce a second capability or let the public API sequence internal durable states.

## Required architecture boundary

```text
browser
  -> optional edge admission / rate limit
  -> narrow Rust sandbox API
  -> durable run record and bounded scheduler
  -> isolated fixed Kubernetes demonstration target
  -> Kapsel execute / reconcile
  -> append-only public event projection and signed receipt
```

- Keep the static website and the native runner independently deployable.
- Run the native Rust service only on infrastructure that explicitly supports its process,
  filesystem, SQLite, network, and Kubernetes requirements.
- Do not assume a Worker isolate can execute the native Kapsel binary or that a container platform
  supports privileged nested `kind` execution.
- Prefer a dedicated sandbox cluster with one owned namespace and narrowly scoped service account
  per run over Docker-in-Docker or visitor-provided clusters.
- An edge Worker may admit, rate-limit, and proxy requests; it must not become the operation journal
  or source of receiver truth.

## Public input boundary

The first API accepts only a fixed scenario selector such as healthy rollout or intentionally
unavailable image. The server owns namespace, Deployment, container, immutable image digest, grant,
trust roots, signing material, Kubernetes authority, paths, deadlines, and cleanup policy.

No caller-supplied shell, command, manifest, patch, namespace, resource name, image, digest,
registry, grant, public key, credential, cluster, callback URL, lifecycle action, or receipt
classification is accepted.

## Work

1. Add a canonical sandbox API and deployment contract before implementation. Define request,
   operation identity, accepted/terminal states, event ordering, reconnect/replay behavior, receipt
   retrieval, expiry, errors, cancellation policy, and versioning limits.
2. Extend the threat and privacy owners for public abuse, denial of service, enumeration, event and
   receipt disclosure, log redaction, artifact retention, key custody, runner compromise, namespace
   escape, cleanup failure, and dependency or image compromise.
3. Build one narrow Rust service adapter over the existing `Application`; keep HTTP, admission,
   scheduling, and public projections outside the deep effect-gateway module.
4. Persist the run before dispatch and make status reconnectable. Browser disconnect, timeout, or
   failed event streaming must not cancel the operation or become `FAILED`.
5. Provision one uniquely owned sandbox target per run with fixed images, resource quotas, network
   restrictions, deadline, concurrency cap, global kill switch, and ownership-safe forced cleanup.
6. Project only a bounded public timeline: admission, durable attempt, provider seam crossing,
   process termination when exercised, recovery, receiver observation, classification, publication,
   and cleanup. Never expose secrets, raw credentials, internal paths, or uncontrolled provider
   logs.
7. Prove healthy and unavailable-image paths, process restart, stream reconnect, duplicate client
   submission behavior, saturation, timeout, cleanup failure, and runner restart.
8. Rehearse deployment and rollback from an exact clean revision. Record service revision, Kapsel
   revision, fixed images, public API version, environment, and residual risks.

## Acceptance

- Only the two documented fixed scenarios can reach execution.
- Every admitted run has a durable identity, immutable server-owned authority inputs, bounded
  resources, deadline, retention period, and cleanup owner before provider mutation.
- Repeated or concurrent browser requests cannot cause an unbounded run count or silently create a
  second mutation for one admitted operation.
- A disconnected client can recover the same ordered event projection and terminal receipt without
  controlling reconciliation.
- `SUCCEEDED`, `FAILED`, and `UNKNOWN` preserve the existing receiver-bounded meaning;
  infrastructure or stream errors are not promoted into provider outcomes.
- Public receipts and events pass a disclosure review and contain no credentials, signing material,
  private infrastructure identifiers, or unnecessary visitor data.
- Namespace and resource ownership prevent one run from observing, mutating, or cleaning another.
- Saturation fails closed before dispatch with a stable retryable response. A kill switch prevents
  new admission without corrupting already durable runs.
- Forced cleanup is observable and retryable but remains distinct from the Kapsel operation receipt.
- Deterministic service tests, Kapsel's repository gate, a live sandbox integration gate, and an
  explicit security review pass before public traffic.

## Non-goals

No arbitrary Kubernetes API, production cluster, customer credential, user account, grant editor,
operator console, fleet dashboard, generic logs product, billing, generic provider SDK, second
capability, or production-readiness claim.

## Next action

Complete KAP-0049, then write the sandbox API, deployment, threat, privacy, and test owners before
choosing the Rust HTTP framework or hosting provider.

## Residual risk

A public sandbox creates real abuse, isolation, availability, disclosure, and operating-cost risk.
Passing this packet proves only one controlled demonstration path; it does not prove safe production
operation, multi-tenancy, customer-hosted integration, or a commercially viable cloud service.
