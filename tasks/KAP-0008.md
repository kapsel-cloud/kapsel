# Implement Profile-Neutral SQLite Acceptance

Problem: Local atomic acceptance and submission recovery must work for any supported profile without becoming an arbitrary event lake or confusing operational attempts with remote facts.

Acceptance:

- Store profile ID/version, source ID, idempotency key, finalized signed statement bytes/hash, public evidence state, internal sync state, acknowledged submission ID, receipt bytes, and timestamps.
- Accept only a finalized, signed statement produced from typed evidence validated by a registered `EvidenceEncoder`; reject arbitrary event payloads.
- Atomically commit signed bytes plus durable submission outbox.
- Use public states `accepted`, `submitted`, `witnessed`, and `rejected`; enter `submitted` only after backend acknowledgement/submission ID.
- Keep pending, in-flight, ambiguous, retryable, acknowledged, and terminal as separate internal sync states.
- A timeout or attempted request remains publicly `accepted` with internal ambiguous/retryable state.
- Document WAL/FULL and host/filesystem/SQLite durability assumptions.
- Support explicit deletion of profile-owned disclosed material while retaining configured commitments/receipts; report deleted disclosure honestly.
- Do not build scheduling or a general retention-policy engine.
- Keep domain fields out of generic tables; use opaque versioned profile material where required.

Verification:

- Crash/restart and fault injection yield old-valid or new-valid state, never partial acceptance.
- Remote ambiguity never produces public `submitted` without acknowledgement.
- Two compiled profiles use the same schema and acceptance machinery.
- Storage review finds no profile-specific columns or generic arbitrary-event API.
