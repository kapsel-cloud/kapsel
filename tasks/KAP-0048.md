# Decide Whether To Add A Crates.io Distribution

Status: conditional; blocked by approved evidence that Cargo or Rust-library distribution is useful.

Problem: crates.io contains historical `0.1.0-alpha.1` metadata, while the supported `0.1.0`
distribution is the GitHub Release artifact. Publishing merely to make registry metadata look
current would add a distribution and support surface without evaluator evidence.

Contract owners: the [technical scope](../docs/SCOPE.md), [release contract](../docs/RELEASE.md),
[build guide](../docs/BUILD.md), [KAP-0045 release decision](KAP-0045.md), and
[KAP-0046 direction decision](KAP-0046.md).

Depends on: approved aggregate evidence from [KAP-0047](KAP-0047.md) and an explicit maintainer
decision that package-registry distribution belongs to the selected technical route.

## Current facts

- The published `v0.1.0` contract explicitly excludes a separate package-registry artifact.
- Published crates.io version metadata is immutable; the historical alpha names
  `github.com/mitander/kapsel`, which redirects to `kapsel-cloud/kapsel`.
- Current source metadata, Git remote, GitHub repository, and project links already use
  `kapsel-cloud/kapsel`.
- `mitander` remains the crates.io owner. A GitHub organization team is optional administrative
  redundancy, not a prerequisite for correcting repository ownership.

## Decision rule

Do not publish crates.io `0.1.0` retroactively. If evidence selects Cargo installation or Rust
library consumption, create a new patch release such as `0.1.1` or later and own the registry as an
explicit distribution channel. Otherwise retain the historical alpha without yanking it solely for
metadata or presentation reasons.

## Work if unblocked

- Name whether the supported registry use is library consumption, `cargo install`, or both.
- Define supported targets and repeat the non-compatibility, security, and production-readiness
  limits without implying that crates.io broadens platform support automatically.
- Audit exact package contents and exclude development-only or private material.
- Test `cargo package --locked`, `cargo publish --locked --dry-run`, clean installation or
  dependency use, generated rustdoc, docs.rs expectations, and the relevant evaluator path.
- Produce the GitHub tag, release notes, package source, and crates.io upload from one exact clean
  revision with consistent version and repository metadata.
- Record package checksum, registry URL, docs URL, source revision, commands, and residual risks.

## Acceptance

- Approved evaluator evidence explains why registry distribution exists.
- The release owner names crates.io and docs.rs as deliberate surfaces and states what they do not
  support.
- The new version never moves or reuses `v0.1.0` and never publishes a different source revision
  under the already released `0.1.0` identity.
- Package inspection, clean-use tests, default repository gates, and the narrowest applicable
  release gates pass before publication.
- Existing named ownership may remain with `mitander`; any team-owner addition follows Cargo's
  restricted GitHub-team owner model and is reviewed separately.

## Non-goals

No publication for badge freshness, download counts, search ranking, old-link cosmetics, or an
unstated Rust API compatibility promise.

## Next action

Remain blocked. Revisit only if KAP-0047 produces approved evidence for Cargo or Rust-library use.
