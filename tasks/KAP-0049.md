# Harden The Self-Serve Local Alpha

Status: complete; `v0.1.1` published and public assets verified.

Problem: `0.1.0` proves the experiment, but the published evaluator path asks a curious practitioner
to assemble too much of the experience before the crash-recovery proposition becomes legible. The
existing capability needs one bounded self-serve path before broader technical distribution can
produce useful independent evidence.

Contract owners: the [technical scope](../docs/SCOPE.md), [evaluator guide](../docs/EVALUATOR.md),
[command contract](../docs/COMMANDS.md), [MCP adapter contract](../docs/MCP.md),
[release contract](../docs/RELEASE.md), and
[active experiment boundary](../docs/experiments/KAP-0038-kubernetes-effect-gateway-boundary.md).

Depends on: the completed [KAP-0045 release](KAP-0045.md). Unblocks the public sandbox packet in
[KAP-0050](KAP-0050.md) and contributes to the broader evaluator evidence cycle in
[KAP-0047](KAP-0047.md). It does not unblock [KAP-0046](KAP-0046.md).

## Result

Starting from a clean supported x86-64 GNU/Linux machine with the documented prerequisites, a
developer can use an exact published artifact to reach and understand the real disposable-`kind`
crash-recovery receipt within ten minutes and without maintainer assistance.

The local CLI, fixed-schema stdio MCP adapter, and evaluator demonstration remain three narrow
surfaces over the same Rust `Application` behavior. This packet improves their self-serve path; it
does not create a new product model or compatibility promise.

## Published 0.1.0 baseline

Measured 2026-07-20 before implementation changes. The clean evaluator was an isolated OrbStack
x86-64 Debian 12 VM (`x86_64`, glibc 2.36) with Docker 20.10.24 on an amd64 daemon, kind 0.32.0,
kubectl 1.33.9, and Python 3.11.2. No source checkout was mounted and `kind get clusters` reported
no clusters. Prerequisite installation time was excluded.

Starting in an empty evaluation directory, the evaluator had to infer the first two commands because
the guide begins from a directory already containing the assets:

```sh
curl -fLO https://github.com/kapsel-cloud/kapsel/releases/download/v0.1.0/kapsel-0.1.0-x86_64-unknown-linux-gnu.tar.gz
curl -fLO https://github.com/kapsel-cloud/kapsel/releases/download/v0.1.0/kapsel-0.1.0-x86_64-unknown-linux-gnu.tar.gz.sha256
archive=kapsel-0.1.0-x86_64-unknown-linux-gnu.tar.gz
sha256sum --check "$archive.sha256"
python3 - "$archive" <<'PY'
# Exact 58-line bounded extraction program from docs/EVALUATOR.md.
PY
cd "$(basename "$archive" .tar.gz)"
python3 -m json.tool RELEASE-METADATA.json
install -d "$HOME/.local/bin"
install -m 0755 bin/kapsel "$HOME/.local/bin/kapsel"
export PATH="$HOME/.local/bin:$PATH"
command -v kapsel
KAPSEL_DEMO_EXECUTABLE="$PWD/libexec/kapsel-demo-harness" \
KAPSEL_DEMO_ASSET_DIRECTORY="$PWD/share/kapsel" \
  "$PWD/share/kapsel/demo-kind-crash-recovery.sh"
```

Download, checksum, bounded extraction, metadata inspection, and user-local installation took two
seconds. The primary demonstration then failed during kind node preparation after 60 seconds with
`could not find a log line that matches "Reached target .*Multi-User System.*|detected cgroup v1"`.
The full published path therefore produced no receipt after 62 seconds. A diagnostic-only retry
using Docker's `cgroupfs` driver failed at the same point after ten seconds; that daemon change is
not part of the published path and does not alter the baseline result.

Observed friction and comprehension problems:

- the evaluator must navigate from the release page to the guide and invent the asset-download
  commands before following the documented path;
- verification requires copying a long embedded Python extractor, while installation is not used by
  the separate demonstration executable;
- the primary command requires two internal layout environment variables and a long path instead of
  locating its fixed packaged assets;
- prerequisite output does not summarize detected versions or Docker compatibility, and the kind
  failure gives no concrete compatibility check or corrective action;
- phase labels imply two process kills and restarts, but output before the final report does not say
  that `apply_started` was durable, distinguish reconciliation from a blind second mutation, or
  explain the receiver-bounded result and `UNKNOWN` limit;
- the failed run named a retained bounded-log directory and attempted deletion of its unique owned
  cluster. A following `kind get clusters` confirmed no cluster remained. The hidden host workspace
  was removed, but the command emitted no workspace-cleanup confirmation or copyable retry command;
  and
- because no receipt was reached, the process-kill, restart, mutation-count, receiver disposition,
  frozen receipt, and offline inspection path were not independently understandable from this run.

This baseline does not satisfy the ten-minute receipt gate. It establishes both avoidable front-door
friction and an undeclared Docker/kind host-compatibility failure that a self-serve path must
diagnose rather than presenting generic kind output.

## Acceptance

- The timed clean-environment run reaches a receipt in at most ten minutes after documented
  prerequisites are present.
- The primary path requires no source checkout, private path, undocumented fixture, or maintainer
  intervention.
- The output makes the process kill, restart, no-blind-second-mutation boundary, receiver outcome,
  and explicit `UNKNOWN` limit distinguishable without reading source code.
- Healthy and unavailable-image demonstrations preserve the existing authorization, lifecycle,
  Kubernetes, and receipt contracts.
- The MCP example exposes only `kubernetes.set_deployment_image` request fields and does not let the
  caller choose operator authority or internal lifecycle actions.
- Missing prerequisites and cleanup failures identify a concrete corrective action and do not
  silently weaken ownership checks.
- The narrow command, MCP, demo-harness, release-artifact, and repository gates pass as applicable;
  the live-kind rehearsal is recorded separately from deterministic CI.
- KAP-0047 records the exact artifact and environment used when broader evaluation begins.

## Implementation evidence

The published `0.1.1` patch keeps `v0.1.0` immutable. It adds packaged self-location, prerequisite
version diagnostics, elapsed phases, explicit ownership-safe cleanup, the lifecycle evidence
summary, and one generic stdio MCP configuration with a complete fixed-tool request.

Deterministic gates passed on 2026-07-20:

```sh
python3 scripts/test-demo-kind-crash-recovery.py
cargo test --locked --test e2e_mcp_adapter
cargo make test-demo-harness
cargo make test-release-artifact
cargo make test-release-reproducibility
./scripts/ci-local.sh
cargo make fmt-check
cargo make docs-links
cargo make style-audit
git diff --check
```

The dirty-worktree artifact lane assembled and smoke-tested
`kapsel-0.1.1-x86_64-unknown-linux-gnu.tar.gz`; the reproducibility lane produced matching isolated
archives. This is packaging evidence, not publication evidence.

The separate source-built live gate passed on macOS arm64 with Docker 29.4.0, kind 0.32.0, kubectl
1.33, and Python 3.14.6. It reached the classifier-complete `FAILED` receipt and lifecycle summary
in 58 seconds, removed its owned cluster before offline inspection, and removed its owned workspace
on exit.

The first clean x86-64 artifact rehearsal then exposed one hidden `shasum` dependency after the
second process termination. The harness now computes its receipt digest through documented Python
3.11 instead, and a deterministic test makes an available `shasum` fail to prevent regression.

The corrected clean artifact was assembled from revision `0a7788587e894b058d100ee941bd19811e3f6846`
with `source_dirty: false` and checksum
`67ad81831342ef5ebd63b6a1becb489007fe3446afd2b1005237e1280308e43c`. The exact extracted path ran in
a clean QEMU x86-64 Debian 12 VM with Linux 6.1.174, Docker 20.10.24, kind 0.32.0, kubectl 1.33.9,
and Python 3.11.2. Checksum verification, bounded extraction, metadata inspection, and installation
took three seconds. The one-command real-kind demonstration reached and explained the receipt,
removed its cluster and workspace, and completed in 204 seconds total. No source checkout or
internal-layout environment variable entered the VM.

`kapsel-0.1.1-x86_64-unknown-linux-gnu.tar.gz` with SHA-256
`d1a5bfec47012e126a9c8b351fbe2331aa52e12c2df0fd41dd2a91c03b7c7fb4`. Metadata records that exact
revision, `source_dirty: false`, the sole supported target, license, and both binary digests.

The final deterministic, MCP, demo-harness, repository, artifact-smoke, and byte-for-byte
reproducibility gates passed at that revision. An independent acceptance review found no blocker.
The exact archive then passed from a separate clean QEMU x86-64 Debian 12 machine with Linux
6.1.174, Docker 20.10.24, kind 0.32.0, kubectl 1.33.9, and Python 3.11.2. Checksum verification,
bounded extraction, metadata inspection, and installation took four seconds. The extracted
one-command path completed in 381 seconds, reported one apply, the two exact process terminations,
restart-only reconciliation, `SUCCEEDED` and `ProgressDeadlineExceeded`-owned `FAILED` outcomes,
frozen receipt bytes, offline `INSPECTED`, and the explicit `UNKNOWN` boundary, then removed its
cluster and workspace. No source checkout or internal-layout environment variable entered the VM.

One other clean QEMU run reached a healthy-fixture progress deadline under slow emulation and exited
with ownership-safe cluster and workspace cleanup. A warm diagnostic retry passed, and the separate
clean 381-second run above supplied acceptance evidence. The transient is retained as a
self-serve-reliability residual risk rather than a receiver result.
