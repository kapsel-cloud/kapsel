# Freeze Profile-Oriented V1 Charter

Problem: Kapsel must remain a witness and verification layer for selected consequential actions; one reference workflow must not become the company or leak into the proof core.

Acceptance:

- Define the thesis, claims/non-claims, V1 completion criterion, and non-goals.
- Define strict core -> profile -> adapter dependency direction and a compile-time composition root.
- Define profiles as typed, versioned evidence contracts—not arbitrary event payloads.
- State that technical V1 uses `kapsel.reference-action.v1` by default; commercial profile selection is independent.
- State standards posture: one SCITT statement profile, receipt profile, algorithm suite, and backend.
- Freeze the producer path: finalize typed evidence -> profile validate/encode -> sign one statement -> atomically accept signed bytes plus outbox -> submit -> persist receipt.
- Define public evidence states as `accepted`, `submitted`, `witnessed`, and `rejected`; `submitted` requires backend acknowledgement/submission ID.
- Define internal sync states separately: pending, in-flight, ambiguous, retryable, acknowledged, terminal.
- State that timeout/attempt alone remains publicly `accepted`, never `submitted`.
- State that V1 does not guarantee authorization was witnessed before action execution; adapter journaling, stable correlation/idempotency, outcome observation, and reconciliation protect the action-to-evidence transition under documented assumptions.
- Keep multiple linked Kapsel statements, pre-action witnessing, generic tracing, dashboards, search, SIEM, policy language, and a Kapsel-operated witness out of V1.

Verification:

- Core owner docs contain no workflow-specific domain types or product positioning.
- Technical V1, reference profile, and commercial-validation gates are separate.
- Architecture review finds no profile-to-core inversion or contradictory lifecycle ordering.
