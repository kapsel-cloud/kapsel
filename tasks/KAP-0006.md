# Implement Offline Verifier Core

Problem: The verifier is the trust substrate and must validate hostile packages without network access or domain concepts in its engine.

Acceptance:

- Define a narrow `ProfileValidator` interface in the verifier boundary for profile identification, collateral validation, linkage checks, disclosure checks, and bounded report output.
- Verify in order: safe parse -> one-statement/cardinality structure -> statement/receipt cryptography -> external trust policy -> profile dispatch -> disclosure/completeness analysis -> verdict aggregation.
- Enforce file, byte, nesting, record, proof, algorithm, and critical-parameter bounds.
- Reject traversal, duplicate paths/JSON keys, symlinks, bombs, malformed CBOR, downgrade, key-purpose confusion, partial ambiguity, and conflicting claims.
- Treat package keys/certificates/metadata as untrusted collateral unless anchored by external policy.
- Register validators through an explicit composition API; do not use runtime plugin loading.
- Distinguish authentic, trusted, linked, externally witnessed, disclosed, omitted/deleted, and unproven claims.
- Implement the KAP-0004 verdict aggregation exactly and make no network calls.

Verification:

- Static fixture matrix and hostile corpus pass.
- Independent review finds no blocker/high trust or parser issue.
- Adding a compiled profile requires registration but no verifier-engine modification.
