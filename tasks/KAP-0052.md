# Build The Sandbox Service Package

Status: complete; deterministic acceptance accepted on 2026-07-21. KAP-0053 is next queued.

Problem: accepted sandbox contracts need one independently deployable Rust implementation without
coupling HTTP, anonymous admission, capacity, retention, public projections, or cleanup to the deep
effect-gateway module.

Direct owners: [V1 technical direction](../docs/VISION.md), [architecture](../docs/ARCHITECTURE.md),
the sandbox API and deployment owners produced by [KAP-0051](KAP-0051.md), and
[KAP-0050](KAP-0050.md).

Depends on: [KAP-0051](KAP-0051.md). Feeds [KAP-0053](KAP-0053.md).

## Result

A new `kapsel-sandbox` workspace package depends one-way on `kapsel` and deterministically proves
fixed-scenario admission, scheduling, execution, reconnectable public projection, receipt retrieval,
and cleanup orchestration without a live cluster.

## Package seam

`kapsel-sandbox` owns:

- versioned HTTP translation;
- durable public run identity and duplicate handling;
- capacity admission, deadlines, and global stop;
- bounded scheduling;
- server-owned per-run composition inputs;
- ordered disclosure-reviewed public projection;
- receipt retrieval projection and retention; and
- cleanup scheduling and status distinct from operation outcome.

The root `kapsel` package continues to own authorization, durable effect lifecycle, provider
attempt, recovery, receiver observation, classification, and receipt construction. The sandbox
package must not read or sequence private gateway journal states.

## Work

1. Add only the dependencies required by the accepted contracts and record why the Rust HTTP,
   persistence, and serialization choices fit supported deployment constraints.
2. Implement durable admission before dispatch, opaque bounded run identities, idempotent duplicate
   handling, concurrency caps, deadlines, retry-after behavior, and a global stop.
3. Compose one `Application` per owned run with fixed server-generated authority and target inputs.
4. Build reconnectable projection from public application reports and sandbox-owned events; never
   serialize local receipt paths or internal causes.
5. Implement bounded receipt retrieval, expiry, retention deletion, and disclosure-safe errors.
6. Implement a target-provisioning and cleanup module whose fake adapter can prove cross-run
   ownership, timeout, retry, and forced cleanup deterministically.
7. Add deterministic tests for both scenarios, duplicate clients, disconnect/reconnect, replay,
   saturation, unknown protocol versions, hostile input, runner restart, cleanup failure, and secret
   non-disclosure.
8. Keep the ordinary `kapsel` artifact and default local interfaces free of sandbox HTTP and
   deployment dependencies.

## Acceptance

- The workspace graph contains one one-way `kapsel-sandbox -> kapsel` dependency and no reverse
  dependency.
- Deleting `kapsel-sandbox` leaves the local CLI, MCP adapter, gateway, tests, and release artifact
  behavior unchanged.
- The package uses `Application` rather than reproducing or sequencing gateway lifecycle.
- Admission is durable before dispatch and bounded independently of operation idempotency.
- Reconnect returns the same ordered public projection and terminal receipt identity.
- Saturation and global stop fail closed before provider execution.
- Cleanup failure is observable and retryable without changing `SUCCEEDED`, `FAILED`, or `UNKNOWN`.
- Tests prove no credentials, grants, signing material, private paths, raw provider bodies, or
  unnecessary visitor data enter public output or logs.
- The deterministic workspace gate and focused sandbox contract tests pass without Docker,
  Kubernetes, external network, or the website checkout.

## Implementation evidence

The `crates/kapsel-sandbox` package depends one-way on the root `kapsel` package and leaves the root
artifact dependency tree unchanged. Its narrow service boundary owns strict bounded `v1` HTTP
translation, SQLite admission/projection transactions, OS-random 128-bit run identity, 32 queued and
8 active reservations, an admission-frozen policy revision/inventory, cleanup identity and
180-second duration, a dispatch-established absolute deadline, durable scheduler leases/recovery,
ordered replay, expiry tombstones, pending-owned immutable receipt-file installation serialized with
orphan collection, periodic/restart-before-serve privacy sweeps, and cleanup completion gated by
absence evidence for every append-only durable kind/namespace/name/UID/owner row. Exact per-object
identity/UID/owner/content verification and cross-run UID exclusion fail closed before `Application`
composition or invocation. The package calls `Application::execute` and `Application::reconcile`
with fixed server-derived requests and projects only `OperationReport`.

Dependency choices remain concrete to this package: `http` supplies typed translation without
selecting a server framework; `serde` and `serde_json` own the exact bounded documents; bundled
`rusqlite` owns one deterministic local service store; `getrandom` creates opaque run locators;
`sha2` verifies exact receipt bytes and keys private tombstone digests; and `rustix` checks Unix
owner-private paths. Test-only `kube`, `tower-test`, Ed25519, and Tokio dependencies drive the real
`Application` against deterministic HTTP responses. No dependency or interface moves into `kapsel`.

Focused tests cover durable replay and conflict, queue/active saturation, global stop, strict
hostile HTTP and authority exclusion, both fixed scenarios, fair dispatch, lease exclusion/recovery,
admission-frozen policy/inventory/duration/cleanup facts, queued-age fairness and dispatch-relative
deadlines, missing/stale/permissive/wrong-owner policy mismatch before Application invocation, setup
failure with owned resources and confirmed absence, every-cursor and concurrent-append pagination,
autonomous expiry and tombstone deletion including a direct first restart after both windows, secure
no-follow 0600 database entry handling, and cleanup cross-run/historical/per-object/presence failure
and retry. Hostile HTTP covers POST queries and forwarding, tracing, and both hyphenated and
`clientcert` client-certificate header families. An injected marker-only crash proves reconciliation
submits the same server-owned request when no gateway operation exists. Process cancellation after
one returned mutation proves gateway-state recovery observes without another patch even after the
ordinary absolute deadline event. A receipt-reference trigger proves terminal report durability and
exact immutable object recovery across the object/database publication window. A concurrent
collector test preserves an installed pending-owned object until its exact bytes are reverified and
availability commits, then safely collects stale ownership; terminal and receipt events remain
singular and contiguous. `scripts/test-sandbox-package-boundary.py` checks the graph and compiles
the ordinary root package from a temporary copy with `kapsel-sandbox` deleted.

## Non-goals

No live deployment, browser UI, production API, customer account, billing, arbitrary capability,
shared protocol crate, SDK, public provider trait, storage abstraction, or `kapseld` implementation.

## Acceptance review

Fresh domain and security reviews checked the final package against the sandbox API, deployment,
testing, threat, and privacy owners. Findings about admission-frozen policy evidence, scheduler
leases, dispatch-relative deadlines, pre-submission and post-mutation recovery, no-resource cleanup,
receipt publication/orphan races, append-only cleanup ownership, autonomous retention, SQLite path
safety, and hostile routing headers were resolved through the sole repository writer. The focused
sandbox gate and full deterministic repository gate pass on the accepted working tree.

## Residual risk

Deterministic adapters prove orchestration semantics, not real namespace isolation, Kubernetes RBAC,
network containment, cleanup under infrastructure failure, public abuse resistance, or operating
cost.
